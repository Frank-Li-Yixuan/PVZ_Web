# 双人合作枪械英雄塔防游戏 V0.1 总玩法设计文档

## 0. 文档目的

本文档是 V0.1 版本的**总玩法设计母版**，用于定义游戏的核心体验、玩法结构、系统边界、规则关系和版本范围。

本文档不负责详细代码架构、不负责具体网络协议、不负责完整美术生成提示词、不负责音频资源清单。这些内容将拆分到后续文件：

| 文件 | 责任 |
|---|---|
| `GAME_DESIGN_V0_1.md` | 总玩法规则、系统边界、核心循环 |
| `COMBAT_NUMBERS_V0_1.md` | 具体血量、伤害、速度、波次、Boss 数值 |
| `NETWORK_SYNC_SPEC_V0_1.md` | 客户端/服务器同步协议与权威状态规则 |
| `ART_DIRECTION_V0_1.md` | 美术风格、角色视觉方向、image-gen 资源生成规范 |
| `AUDIO_PIPELINE_V0_1.md` | 免费音频资源、许可、命名、接入流程 |
| `CODEX_IMPLEMENTATION_PLAN.md` | Codex 分阶段实现计划与验收标准 |

V0.1 的目标是做出一个**可玩的垂直切片 Demo**，验证玩法是否成立，而不是一次性做成完整商业游戏。

## 1. 游戏一句话定义

**一款双人合作 2.5D 枪械英雄塔防游戏。两名玩家共同防守基地，共享阳光资源，种植植物建立防线，操控持枪英雄机动救场，并在每关最后合作击败 Boss。**

---

## 2. 核心设计支柱

V0.1 必须围绕以下五个设计支柱展开。

### 2.1 双人合作，而不是两个单人玩家并排玩

游戏必须让两名玩家产生真实合作：

- 共享阳光；
- 共同决定资源分配；
- 一人可以偏种植防线，另一人偏英雄救场；
- Boss 机制需要两名玩家共同处理；
- 任意一人乱花资源都会影响全队。

设计目标：玩家要经常产生类似这样的交流：

> “先别买子弹，补一个坚果。”  
> “Boss 快冲锋了，留子弹打断。”  
> “我去下路救场，你负责中路种豌豆。”

### 2.2 植物是主防线

植物是游戏的稳定输出与防守核心。英雄不能完全替代植物。

植物负责：

- 长时间持续输出；
- 阻挡路线；
- 生产经济；
- 维持基础防线。

如果玩家只依靠英雄射击，不建立植物防线，应该很难通关。

### 2.3 英雄是机动救场单位

英雄负责处理植物难以处理的紧急情况。

英雄负责：

- 击杀漏网僵尸；
- 处理快速僵尸；
- 对高血量敌人补伤害；
- Boss 战打弱点；
- Boss 读条打断；
- 紧急救线。

英雄不应该成为无限火力单位，因此必须受到弹药限制。

### 2.4 阳光是团队矛盾核心

阳光不是单纯建造资源，而是团队共同决策资源。

阳光用途包括：

- 种植物；
- 买子弹；
- 英雄进化；
- 后续版本可用于修复植物或团队技能。

设计目标：每次消费阳光都应该让玩家产生取舍。

### 2.5 Boss 是合作压力测试器

每关最后 Boss 不能只是血厚怪，而应该检验整局的合作成果：

- 防线是否足够稳；
- 英雄是否保留弹药；
- 阳光是否合理投资；
- 两名玩家是否能分工处理小怪与 Boss 技能；
- 是否能打断关键技能。

## 3. V0.1 版本目标

V0.1 的目标是完成一个 5 到 8 分钟的可玩关卡。

玩家从进入房间开始，经历准备、建造、防守、进化、Boss 战、结算的完整流程。

### 3.1 V0.1 必须验证的体验

| 体验目标 | 验证方式 |
|---|---|
| 双人合作成立 | 两名玩家共享阳光并共同防守 |
| 枪械英雄有价值 | 英雄能救场、打弱点、打断 Boss |
| 子弹限制成立 | 玩家不能无脑开火，必须买弹药或节省弹药 |
| 植物防线成立 | 不种植物会明显难以通关 |
| Boss 战有高潮 | Boss 至少有两个阶段和可打断技能 |
| Codex 可持续开发 | 系统边界清楚，后续可扩展 |

### 3.2 V0.1 不追求的目标

V0.1 暂时不追求：

- 完整商业美术品质；
- 大量植物；
- 多英雄职业；
- 多地图；
- 局外成长；
- 排位匹配；
- 账号系统；
- 复杂剧情；
- 商店、皮肤、付费系统；
- Steam 级别发布质量。

## 4. 推荐项目代号与原创方向

为了避免做成直接模仿经典塔防，本项目需要从命名、视觉、敌人设计上保持原创。

### 4.1 暂定项目代号

可以使用以下工作代号之一：

| 代号 | 气质 |
|---|---|
| `Sunlock Defense` | 阳光、锁线、防守 |
| `Bloomfront Duo` | 双人、植物前线 |
| `Garden Last Stand` | 花园末日防守 |
| `Sprout & Steel` | 植物与枪械并存 |
| `Greenline Guardians` | 五路防线守护者 |

V0.1 文档内部可以暂用：

> **Project Codename: Sprout & Steel**

### 4.2 原创世界观方向

世界不是传统“僵尸入侵花园”，而是：

> 一场生态污染灾变后，变异植物成为最后的防御资源。两名园区守卫使用枪械和培育装置，保护核心温室不被污染生物摧毁。

这种设定允许我们保留：

- 植物防御；
- 变异敌人；
- 阳光资源；
- 枪械英雄；
- Boss 入侵；

同时避免直接复刻已有 IP。

## 5. 核心游戏循环

### 5.1 单局主循环

```text
创建 / 加入双人房间
  ↓
两名玩家准备
  ↓
进入战斗地图
  ↓
开局获得共享阳光
  ↓
种植经济植物建立阳光来源
  ↓
僵尸类敌人开始多波进攻
  ↓
玩家在种植物、买子弹、保留阳光之间取舍
  ↓
中期开放英雄进化
  ↓
压力波消耗玩家资源
  ↓
Boss 前准备
  ↓
两阶段 Boss 战
  ↓
击败 Boss 胜利 / 基地生命归零失败
  ↓
结算统计
```

### 5.2 玩家每 10 秒的短循环

玩家在战斗中不断重复以下行为：

```text
观察哪条路线危险
  ↓
判断是否需要种植物 / 补子弹 / 移动救场
  ↓
执行操作
  ↓
与队友协调资源使用
  ↓
处理新刷出的敌人或 Boss 技能
```

### 5.3 团队资源循环

```text
向日葵产出阳光
  ↓
击杀敌人有概率掉落阳光
  ↓
共享阳光增加
  ↓
玩家消费阳光
  ↓
消费方向影响战斗能力
  ↓
战斗表现反过来决定后续资源压力
```

## 6. 玩家数量与合作模式

### 6.1 玩家数量

V0.1 固定为 **2 名玩家合作**。

不设计 1 人模式，不设计 3 人或 4 人模式。后续版本可以扩展，但 V0.1 不做。

### 6.2 两名玩家的权限

两名玩家权限相同：

- 都能移动；
- 都能射击；
- 都能种植物；
- 都能购买弹药；
- 都能选择英雄进化；
- 都能触发准备状态。

### 6.3 两名玩家的差异

V0.1 不做职业差异，但通过进化路线制造分工：

- 玩家 A 可以选择火力路线；
- 玩家 B 可以选择支援路线；
- 或者两人都选择控制路线；
- 允许自由组合。

设计目标：系统不强制分工，但鼓励分工。

## 7. 地图与空间设计

### 7.1 地图结构

V0.1 使用五路横向防守结构。

```text
敌人出生侧                         基地侧
Lane 1: [ ][ ][ ][ ][ ][ ][ ]  →  Base
Lane 2: [ ][ ][ ][ ][ ][ ][ ]  →  Base
Lane 3: [ ][ ][ ][ ][ ][ ][ ]  →  Base
Lane 4: [ ][ ][ ][ ][ ][ ][ ]  →  Base
Lane 5: [ ][ ][ ][ ][ ][ ][ ]  →  Base
```

### 7.2 路线数量

固定 5 条路线。

原因：

- 5 路足够制造多线压力；
- 2 名玩家需要移动补位；
- 比 3 路更有塔防味；
- 比 6 路以上更容易控制 V0.1 难度。

### 7.3 种植格

每条路线有 7 个可种植格。

总种植格数量：

```text
5 lanes × 7 columns = 35 plant cells
```

### 7.4 英雄移动空间

英雄不是格子单位，英雄可以在地图上自由移动。

英雄移动规则：

- 可以跨路线移动；
- 可以靠近格子进行种植；
- 可以站在植物附近；
- 可以对鼠标方向射击；
- 不能走出地图边界；
- V0.1 暂不做复杂碰撞体积。

### 7.5 敌人移动规则

敌人是路线单位。

敌人规则：

- 从敌人出生侧生成；
- 生成后绑定一条路线；
- 沿路线向基地移动；
- 默认不换线；
- 遇到植物后停下攻击；
- 到达基地后造成基地伤害。

## 8. 核心资源：共享阳光

### 8.1 阳光定义

阳光是全队共享资源，不属于个人玩家。

屏幕上显示一个统一的队伍阳光数值。

### 8.2 阳光主要来源

V0.1 阳光来源：

| 来源 | 说明 |
|---|---|
| 初始阳光 | 开局给队伍一笔启动资源 |
| 经济植物产出 | 周期性为队伍提供阳光 |
| 敌人掉落 | 击杀敌人有概率给阳光 |
| Boss 阶段奖励 | Boss 进入二阶段时给少量阳光 |

### 8.3 阳光主要用途

| 用途 | 设计目的 |
|---|---|
| 种经济植物 | 投资后续资源 |
| 种攻击植物 | 提升持续输出 |
| 种防御植物 | 拖延敌人推进 |
| 购买弹药 | 让英雄能继续救场 |
| 英雄进化 | 让玩家获得局内成长 |

### 8.4 共享阳光的设计风险

共享阳光容易带来一个问题：

> 一个玩家乱买子弹，另一个玩家无法种植物。

V0.1 不取消共享资源，而是通过以下规则限制滥用：

- 购买弹药有个人冷却；
- 备弹有上限；
- 英雄进化成本高；
- 所有消费都必须服务器校验；
- UI 明确显示谁消耗了阳光。

### 8.5 阳光体验目标

阳光不应该一直过剩，也不应该一直缺到无法操作。

理想状态：

- 前期：玩家想多种向日葵，但有防守压力；
- 中期：玩家开始犹豫是补植物还是进化英雄；
- Boss 前：玩家要决定买弹药还是补防线；
- Boss 中：阳光短缺会带来紧张感。

## 9. 植物系统总设计

### 9.1 植物定位

植物是防线核心，承担稳定功能。

V0.1 只做三种植物。

| 植物 | 定位 | 核心价值 |
|---|---|---|
| 日光芽 / Sunbloom | 经济 | 产出阳光 |
| 豆荚炮 / Peashotter | 输出 | 持续攻击本路敌人 |
| 木壳墙 / Barkwall | 防御 | 阻挡敌人推进 |

名称可以继续调整，但必须保持原创，不直接使用已有 IP 名称。

### 9.2 经济植物

经济植物负责阳光产出。

设计目标：

- 诱导玩家前期投资经济；
- 被敌人攻击时会造成经济损失；
- 玩家需要保护经济植物。

### 9.3 输出植物

输出植物负责本路线持续火力。

设计目标：

- 让防线具备自动攻击能力；
- 迫使玩家考虑每条路线的输出分布；
- 不让英雄成为唯一输出。

### 9.4 防御植物

防御植物负责拖延敌人。

设计目标：

- 给输出植物争取时间；
- 给英雄救场争取时间；
- 在 Boss 战中成为关键缓冲。

### 9.5 种植规则

玩家种植必须满足：

1. 目标格子为空；
2. 目标格子属于可种植区域；
3. 玩家距离目标格子足够近；
4. 队伍阳光足够；
5. 目标格子没有被敌人占据；
6. 服务器确认请求合法。

### 9.6 植物死亡规则

植物生命值归零后死亡，格子释放。

死亡后：

- 不返还阳光；
- 不保留生产计时；
- 敌人继续向基地推进。

## 10. 英雄系统总设计

### 10.1 英雄定位

英雄是玩家直接操控的机动单位。

英雄核心体验：

- 移动；
- 瞄准；
- 射击；
- 换弹；
- 买子弹；
- 选择进化；
- 救场；
- 打 Boss 弱点。

### 10.2 英雄基础能力

V0.1 两名玩家使用相同基础英雄模板。

英雄具备：

- 生命值；
- 移动速度；
- 手枪；
- 弹匣；
- 备弹；
- 换弹时间；
- 进化状态。

### 10.3 英雄不能承担的职责

英雄不能成为：

- 无限输出炮台；
- 全图清怪单位；
- 完全替代植物的主防线；
- 不受资源制约的强力角色。

### 10.4 英雄死亡与复活

V0.1 使用简单复活机制：

- 英雄生命值归零后进入短暂死亡状态；
- 死亡期间不能移动、射击、种植、买弹药；
- 数秒后在基地附近复活；
- V0.1 暂不做队友扶起机制。

这样可以避免第一次版本因为复杂救援机制扩大范围。

## 11. 枪械与弹药系统总设计

### 11.1 为什么英雄使用枪械

枪械让英雄操作更直接，也更适合 Web 版本：

- 鼠标瞄准自然；
- 子弹数量容易显示；
- 命中反馈清晰；
- Boss 弱点机制容易设计；
- 弹药限制可以防止英雄过强。

### 11.2 初始武器

V0.1 只做一种武器：手枪。

手枪特点：

- 单发伤害中等；
- 射速中等；
- 弹匣较小；
- 需要频繁换弹；
- 适合打弱点和救场。

### 11.3 弹药结构

英雄弹药分两层：

| 弹药类型 | 说明 |
|---|---|
| 弹匣 ammoInMagazine | 当前可直接射击的子弹 |
| 备弹 reserveAmmo | 换弹时补充弹匣的储备 |

### 11.4 换弹规则

换弹时：

- 角色不能射击；
- 持续一段换弹时间；
- 从备弹转移子弹到弹匣；
- 备弹不足时只补充可用数量。

### 11.5 买弹药规则

买弹药消耗共享阳光。

设计目的：

- 让英雄火力和团队经济绑定；
- 让玩家在防线和英雄之间取舍；
- 让 Boss 战产生资源压力。

必须限制：

- 每名玩家买弹药有冷却；
- 备弹不能超过上限；
- 阳光不足不能购买。

## 12. 敌人系统总设计

### 12.1 敌人定位

V0.1 敌人是对防线的压力来源。

敌人不需要复杂 AI，但必须形成三种压力：

| 敌人类型 | 压力类型 |
|---|---|
| 普通敌人 | 基础持续压力 |
| 快速敌人 | 英雄救场压力 |
| 装甲敌人 | 防线火力检查 |

### 12.2 普通敌人

普通敌人用于构成大部分波次。

设计目标：

- 让玩家学习基础防守；
- 让输出植物有明确目标；
- 让击杀掉落阳光形成反馈。

### 12.3 快速敌人

快速敌人用于打破静态防线。

设计目标：

- 迫使英雄移动补位；
- 让玩家不能只看一条路线；
- 让子弹有救场价值。

### 12.4 装甲敌人

装甲敌人用于检查火力和防御。

设计目标：

- 迫使玩家提前种输出植物；
- 让防御植物有价值；
- 让英雄补伤害有意义。

### 12.5 敌人攻击规则

敌人遇到植物后：

- 停止前进；
- 攻击该植物；
- 植物死亡后继续前进。

敌人到达基地后：

- 对基地造成伤害；
- 该敌人从战场移除。

## 13. 波次系统总设计

### 13.1 波次结构

V0.1 一关由 5 个普通波次 + 1 个 Boss 战组成。

| 阶段 | 作用 |
|---|---|
| Wave 1 | 教会基础防守 |
| Wave 2 | 提高普通压力 |
| Wave 3 | 引入快速敌人，促使英雄救场 |
| Wave 4 | 引入装甲敌人，检查防线火力 |
| Wave 5 | 混合压力波，消耗资源 |
| Boss | 合作高潮与最终检验 |

### 13.2 波次节奏

一局目标时长：5 到 8 分钟。

节奏结构：

- 开局准备：短；
- 前两波：教学与经济建立；
- 中间波：压力开始上升；
- 第五波：消耗玩家资源；
- Boss 前：给玩家短暂准备时间；
- Boss：高压收束。

### 13.3 波次设计原则

每一波都必须有明确目的。

不允许出现“只是数量更多”的无意义波次。

每波应该引导玩家理解一个系统：

- 经济；
- 输出；
- 阻挡；
- 英雄救场；
- 弹药管理；
- Boss 前资源规划。

---

## 14. Boss 系统总设计

### 14.1 Boss 定位

Boss 是一关的高潮，不是普通敌人的放大版。

Boss 必须测试：

- 玩家防线是否成型；
- 英雄是否保留弹药；
- 阳光是否合理使用；
- 两名玩家是否会分工；
- 玩家是否能处理机制。

### 14.2 Boss 基本结构

V0.1 Boss 有两个阶段。

| 阶段 | 设计作用 |
|---|---|
| 阶段 1 | 让玩家适应 Boss 节奏 |
| 阶段 2 | 加入冲锋读条和打断压力 |

### 14.3 Boss 核心机制

Boss 至少包含以下机制：

1. **高生命值**：需要植物和英雄共同输出；
2. **召唤小怪**：迫使玩家分心处理多线压力；
3. **弱点暴露**：鼓励英雄精准射击；
4. **冲锋读条**：需要英雄子弹打断；
5. **阶段转换**：低血量后压力升级。

### 14.4 Boss 弱点设计

Boss 周期性暴露弱点。

弱点作用：

- 英雄子弹命中弱点造成额外效果；
- Boss 二阶段读条期间必须命中弱点才能打断；
- 植物可以持续输出，但不负责打断核心机制。

### 14.5 Boss 与阳光资源的关系

Boss 战中阳光应该变得紧张。

玩家可能要做这些取舍：

- 买弹药打断冲锋；
- 种坚果拖延 Boss；
- 补输出植物加快击杀；
- 保留阳光等待危机。

---

## 15. 英雄进化系统总设计

### 15.1 进化定位

英雄进化是局内一次性成长，不是永久养成。

目的：

- 增加中期决策；
- 强化玩家分工；
- 让阳光出现大额消费选择；
- 增加单局变化。

### 15.2 进化开放时机

V0.1 建议在 Wave 3 后开放进化。

原因：

- 玩家已经理解基础种植；
- 快速敌人开始出现；
- 资源开始紧张；
- 进化选择会影响 Boss 准备。

### 15.3 进化路线

V0.1 有三条路线。

| 路线 | 定位 | 适合玩家 |
|---|---|---|
| 火力 | 提升输出与 Boss 弱点伤害 | 喜欢攻击的玩家 |
| 控制 | 提升减速和打断能力 | 喜欢救场的玩家 |
| 支援 | 降低弹药压力，增强团队收益 | 喜欢辅助的玩家 |

### 15.4 进化设计原则

进化不能过强，不能让玩家完全不依赖植物。

进化应该强化英雄职责，而不是改变游戏类型。

---

## 16. 胜负条件

### 16.1 胜利条件

击败 Boss 后胜利。

胜利应立即进入结算界面。

### 16.2 失败条件

基地生命归零后失败。

基地生命会因为敌人突破防线而降低。

### 16.3 玩家死亡不是失败条件

英雄死亡不会直接失败。

原因：

- 双人合作游戏中个人死亡应制造压力，而不是立刻结束；
- 简单复活机制更适合 V0.1；
- 避免新手体验过于惩罚。

## 17. 单局结算设计

### 17.1 胜利结算信息

胜利后显示：

- 通关时间；
- 剩余基地生命；
- 击杀敌人数；
- Boss 击杀结果；
- 两名玩家伤害；
- 两名玩家射击次数；
- 两名玩家购买弹药次数；
- 团队阳光总获取；
- 团队阳光总消耗；
- 种植植物数量。

### 17.2 失败结算信息

失败后显示：

- 存活时间；
- 失败波次；
- 基地被突破次数；
- 如果进入 Boss 战，显示 Boss 剩余生命；
- 阳光使用统计；
- 敌人突破路线统计。

### 17.3 结算目的

结算不是为了排名，而是帮助玩家理解为什么赢或输。

## 18. 操作设计

### 18.1 推荐键鼠操作

| 操作 | 键位 |
|---|---|
| 移动 | WASD |
| 瞄准 | 鼠标 |
| 射击 | 鼠标左键 |
| 换弹 | R |
| 购买弹药 | Q |
| 选择经济植物 | 1 |
| 选择输出植物 | 2 |
| 选择防御植物 | 3 |

| 打开进化选择 | F |

### 18.2 为什么种植不用左键

射击和种植都用左键会造成误操作。

V0.1 建议：

- 左键固定射击；
- E 键固定种植；
- 数字键选择植物。

这样对 Codex 实现和玩家操作都更清晰。

## 20. 美术设计总方向

详细内容后续放入 `ART_DIRECTION_V0_1.md`。

## 24. 设计成功标准

V0.1 成功不等于内容多，而是核心循环成立。

### 24.1 玩法成功标准

V0.1 完成后，应该满足：

1. 两名玩家能自然分工；
2. 植物和英雄都有存在感；
3. 子弹限制让玩家有紧张感；
4. 阳光消费有明显取舍；
5. Boss 战需要合作而不是单人硬打；
6. 输了以后玩家知道原因；
7. 赢了以后玩家愿意再打一局尝试不同进化路线。

### 24.2 工程成功标准

V0.1 工程上应该满足：

1. 客户端和服务器结构清晰；
2. 规则数据驱动，方便调参；
3. 网络同步边界明确；
4. 资源目录规范；
5. 能稳定完成一整局；
6. Codex 后续能继续扩展，而不是重写。

## 27. 当前版本最终判断

V0.1 的核心不是做一个“植物大战僵尸复刻”，而是验证一个更适合联机的原创结构：

> **植物负责稳定防线，枪械英雄负责机动救场，子弹制造行动成本，阳光制造团队决策，Boss 把合作压力推向高潮。**

只要这个结构在 V0.1 中成立，后续再增加植物、武器、英雄职业、Boss、局外成长和美术包装，项目就有继续发展的基础。
